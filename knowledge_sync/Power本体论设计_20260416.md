# Power本体论设计 - 基于Palantir Ontology最佳实践

## 概述

本文基于Palantir Ontology的最佳实践，为蔚来能源的**电池调度**和**充电业务**设计了完整的本体论模型，将业务流程映射到Palantir的Object/Link/Action/Function四层架构。

## 一、电池调度本体设计

### 1.1 业务场景

**两大核心场景**：
1. **电池故障调度** → 换电站采集遥测+图像 → AI识别亚健康/磕碰 → 诊断 → 转运至服务中心/工厂
2. **电池配置调度** → 预测换电需求 → 结合成本和仓位 → 给出各型号最优配置 → 周期性更新

**两种调度方式**：
| 方式 | 特点 | 适用场景 |
|------|------|----------|
| **物流调度** | 可增减电池块数 | 长距离、跨站点调运、送修 |
| **车辆自运** | 仅改变配比，不改变总数 | 近距离、员工车顺路转运，成本低 |

**三层流程**：感知需求 → 生成全局最优方案 → 拆解任务执行闭环

### 1.2 核心对象分类

#### 资产与地点类
- BatteryPack（电池包）、BatteryModel（电池型号）
- SwapStation（换电站）、ServiceCenter（服务中心）、Factory（工厂）、Warehouse（仓库）
- LogisticsTruck（物流车）、Forklift（叉车）、BatteryCrate（电池木箱）、NioVehicle（蔚来车辆）

#### 状态与分析类
- BatteryTelemetrySnapshot（遥测快照）、BatterySurfaceImageSet（表面图像集）
- BatteryHealthIssue（健康问题）、BatteryDiagnosis（诊断结论）
- StationDemandForecast（需求预测）、StationConfigPlan（配置方案）

#### 调度三层结构
- DispatchDemand（调度需求）→ 统一价值量化
- DispatchPlan（调度方案）→ 全局优化结果
- DispatchTask（调度任务）→ 原子执行单元
- LogisticsTrip（物流行程）、VehicleTransferTrip（车辆转运行程）
- IncentiveRecord（积分奖励记录）

### 1.3 设计要点

**三层流程映射到Ontology**：

1. **感知阶段**：
   - Action：IngestBatteryTelemetry、CaptureBatteryImages
   - Function：DetectBatteryHealthIssue、DiagnoseBatteryIssue
   - 输出：BatteryHealthIssue → DispatchDemand

2. **规划阶段**：
   - Function：NormalizeDemandValue（统一量纲）、ComputeTransferCost（成本估算）、OptimizeDispatchPlan（全局优化）
   - Action：PublishDispatchPlan（发布方案）
   - 支持组合：先用车辆集货，再物流运输，降低成本

3. **执行阶段**：
   - Action拆解到具体操作：PrepareCrateAtStation → LoadBatteryToCrate → LoadCrateToTruck → TruckDepart/Arrive → Unload...
   - 车辆调度：SwapBatteryToVehicle → DriveToTargetStation → SwapBatteryToStation → 生成IncentiveRecord
   - 闭环：UpdateDispatchTaskStatus → AutoComplete → CloseDispatchPlan

## 二、充电业务本体设计

### 2.1 设计原则

遵循Palantir最佳实践：
- **以真实业务实体建Object**
- **以稳定关系建Link**
- **以可执行业务步骤建Action**
- 将决策-履约-支付三个环节串连在同一本体中

### 2.2 核心对象分类

#### 主体与资产
- User（用户）、Vehicle（车辆）、Battery（电池）
- Operator（运营商）、ChargingStation（充电桩）

#### 定价与营销
- TariffPlan（电价方案）、MarketingStrategy（营销策略）

#### 决策与流程
- AppSession（会话/决策上下文）
- ChargingOrder（充电订单）、MeteringRecord（计量记录）
- UserWallet（用户钱包）、OperatorAccount（运营商账户）、PaymentTransaction（支付交易）

### 2.3 关键业务约束

通过本体规则和函数强化：
1. **所有权防越权**：必须沿Link回溯到发起人，保证权限
2. **资源排他性**：同一桩同一时间只能有一个充电订单
3. **状态一致性**：状态机必须和事件流一致
4. **定价追溯**：计费时点必须唯一确定生效Tariff，历史不可篡改
5. **营销叠加**：冲突策略按优先级，规则固化到Function
6. **支付对账**：完整生命周期记录，与第三方对账可追溯

### 2.4 典型字段设计要点

**Battery（电池）**推荐使用时间序列属性：
- soc_ts → SOC随时间变化
- temperature_ts → 温度变化
- chargeCycles_ts → 循环数累积

**ChargingStation（充电桩）**：
- power_ts → 实时功率
- availability_ts → 可用性变化

**ChargingOrder（订单）**：
- metering_ts → 计量快照序列
- 状态机：待启动 → 充电中 → 完成/取消/异常

## 三、设计方法论总结

| Palantir层级 | 充电/电池调度对应 | 作用 |
|-------------|-------------------|------|
| **Object Type** | 电池、站点、订单等实体 | 统一语义，代替零散表结构 |
| **Link Type** | 对象之间关系（如BatteryPack locatedAt SwapStation） | 显式建模关系，支持权限遍历和图导航 |
| **Action Type** | StartCharging、LoadBatteryToCrate等操作 | 原子化业务步骤，保证执行一致性 |
| **Function** | 诊断、优化、计费等复杂逻辑 | 封装可复用计算，供LLM/Agent安全调用 |

**核心优势**：
- 从"表结构"到"业务语义图谱"，更贴近真实世界
- OAG（Ontology Augmented Generation）替代RAG，大幅降低AI幻觉
- 从分析到执行全链路闭环，支持AI Agent直接操作业务
- 内置安全审计，满足强合规行业需求
